Incident intelligence for suspicious commits

Turn one bad hash into a postmortem people actually trust.

gitmortem converts a commit hash into a production-style incident report: root cause, blast radius, timeline, severity, and the never-again checklist. Groq is the default path, but the provider layer stays open to OpenAI, Anthropic, OpenRouter, Ollama, and compatible APIs.

How it works

Git facts first, LLM judgment second

  1. Extract Commit metadata, raw diff, and nearby history come from git directly.
  2. Map Structured diff parsing and dependency walking estimate likely blast radius.
  3. Explain Provider-backed analysis turns those facts into a postmortem narrative.

Install

Pick the path your team will actually use

Start with the Python path if you want the cleanest install. The other methods exist to reduce discovery friction, not to replace the core runtime. 

python -m pip install git+https://github.com/lekhanpro/gitmortem.git
export GROQ_API_KEY=your_key
gitmortem HEAD~1

This works today directly from GitHub. After the first tagged PyPI release, the install line becomes pip install gitmortem. 

curl -fsSL https://raw.githubusercontent.com/lekhanpro/gitmortem/main/install.sh | bash
gitmortem HEAD~1 --no-llm
irm https://raw.githubusercontent.com/lekhanpro/gitmortem/main/install.ps1 | iex
python -m gitmortem HEAD~1
npm install -g github:lekhanpro/gitmortem
gitmortem HEAD~1

Useful in JavaScript-heavy environments. The actual analysis still runs through Python 3.10+ under the hood.

FAQ

Short answers to the questions people ask before they try it

Does it work without an LLM?

Yes. --no-llm renders a deterministic report from git facts, parsed diff details, blast radius heuristics, and surrounding commit history.

Is npm the primary package?

No. The Python package is primary. The npm package exists as a discovery and install wrapper for teams that start from JavaScript tooling.

Can I swap providers later?

Yes. Provider selection is runtime configuration. Groq is the default, but the CLI does not hard-code you into one vendor.

Is it ready for release?

Yes. CI, GitHub Pages deployment, GitHub releases, and PyPI publishing workflow are already in the repo. The remaining operational step is your first version tag and PyPI credential configuration.
